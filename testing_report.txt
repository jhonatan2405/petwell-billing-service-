1. INTRODUCCIÓN
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

El billing-service es el microservicio responsable de la facturación y los
pagos dentro de la plataforma PetWell. Sus funciones principales son:

  • Crear y consultar facturas de atención veterinaria.
  • Iniciar enlaces de pago a través de Bold Checkout.
  • Confirmar pagos manualmente (fallback) o vía webhook de Bold.
  • Gestionar la tabla de precios por clínica.
  • Recibir eventos (webhooks) del proveedor de pagos Bold.

Se implementó una suite completa de pruebas sin modificar ningún archivo
dentro de src/. El objetivo fue validar el comportamiento correcto del
microservicio, detectar errores anticipadamente y garantizar la calidad
del código para entrega académica y productiva.

Un mock es una versión simulada de una dependencia. Se usa para aislar
la lógica que se está probando y evitar conexiones reales.

MOCK 1: InvoiceRepository / PaymentRepository (Supabase)
  Archivo: src/repositories/billing.repository.ts
  Técnica: jest.mock() reemplaza la clase completa con funciones jest.fn().
  Motivo: Evitar cualquier conexión real a Supabase durante los tests.
  Qué simula: create, findById, findByOwner, findByClinic, updateStatus, etc.

MOCK 2: PricingRepository (Supabase)
  Archivo: src/repositories/pricing.repository.ts
  Técnica: jest.mock() con objeto de funciones mock.
  Motivo: Aislar PricingService sin tocar la base de datos.
  Qué simula: findByClinic, upsert.

MOCK 3: global.fetch (Servicios externos)
  Archivo: Global en los unit tests de billing.service y webhook.service.
  Técnica: global.fetch = jest.fn().mockResolvedValue({ ok: true, ... })
  Motivo: El servicio notifica al appointment-service tras confirmar un pago.
  Qué simula: Llamadas HTTP al appointment-service (fire-and-forget).

MOCK 4: WebhookService (en tests de integración de webhook)
  Archivo: src/services/webhook.service.ts
  Técnica: jest.mock() reemplaza toda la clase.
  Motivo: Los tests de ruta verifican el comportamiento del controller HTTP
           (respuesta rápida, headers, status codes) sin ejecutar la lógica
           completa del servicio.
  Qué simula: processEvent → retorna resultados controlados por test.

MOCK 5: env module (en bold.util.test.ts)
  Archivo: src/config/env.ts
  Técnica: jest.mock() con objeto mutable para controlar DISABLE_WEBHOOK_SIGNATURE.
  Motivo: env.ts evalúa las variables de entorno al importarse, por lo que
           mutando process.env en tiempo de ejecución no tiene efecto.

6. USO DE MATCHERS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Un matcher es la forma en que Jest verifica si un resultado es el esperado.

toBe
  Uso: expect(result).toBe(true)
  Significado: Compara valores primitivos de forma exacta (===).
  Ejemplo: expect(res.status).toBe(200) — verifica que el código HTTP es 200.

toEqual
  Uso: expect(result).toEqual(baseInvoice)
  Significado: Compara objetos de forma profunda (estructura y valores).
  Ejemplo: verifica que el objeto retornado tiene todos los campos correctos.

toHaveProperty
  Uso: expect(res.body).toHaveProperty('success', true)
  Significado: Verifica que un objeto tiene una propiedad con el valor dado.
  Ejemplo: valida que la respuesta JSON contiene { success: true }.

toBeDefined
  Uso: expect(result).toBeDefined()
  Significado: Verifica que el valor NO es undefined.
  Ejemplo: confirma que el servicio retornó algún resultado (no vacío).

toMatchObject
  Uso: expect(error).toMatchObject({ statusCode: 403 })
  Significado: Verifica que un objeto contiene al menos esas propiedades.
  Ejemplo: valida que el error lanzado tiene el código de estado correcto.

7. PRUEBAS ASÍNCRONAS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Los servicios del billing-service son completamente asíncronos porque
dependen de bases de datos y llamadas HTTP externas. Todas las pruebas
usan async/await.

async/await
  Cada test está marcado como `async` y usa `await` para esperar la
  resolución de promesas antes de verificar el resultado.
  Ejemplo:
    it('crea una factura', async () => {
        const result = await invoiceService.createInvoice(dto);
        expect(result).toHaveProperty('id');
    });

Caso con .resolves
  Verifica que una promesa se resuelve exitosamente con un valor esperado.
  Ejemplo:
    await expect(paymentService.initiatePayment(...))
        .resolves.toHaveProperty('redirect_url');

Caso con .rejects
  Verifica que una promesa es rechazada con el error esperado.
  Ejemplo:
    await expect(invoiceService.createInvoice({ total_amount: 0, ... }))
        .rejects.toMatchObject({ statusCode: 400 });

Prueba fire-and-forget
  El webhook controller responde HTTP 200 inmediatamente y luego procesa
  el evento en segundo plano. Se verifica que aunque la tarea de fondo
  falle, el response ya fue enviado correctamente.

¿Qué es la cobertura?
  La cobertura de código mide qué porcentaje del código fuente fue ejecutado
  durante las pruebas. Es una métrica de calidad, no de perfección.

Tipos de cobertura medidos:

  STATEMENTS (Sentencias)
    Cada instrucción individual del código.
    Ejemplo: const x = 5; — una sentencia.
    Objetivo: >= 80%

  BRANCHES (Ramas / Bifurcaciones)
    Cada camino posible en estructuras condicionales (if/else, ternarios).
    Ejemplo: if (a > 0) { ... } else { ... } — 2 ramas.
    Es el más difícil de cubrir completamente.
    Objetivo: >= 70%

  FUNCTIONS (Funciones)
    Cada función o método que fue invocado al menos una vez.
    Objetivo: >= 80%

  LINES (Líneas)
    Cada línea de código ejecutada.
    Objetivo: >= 80%

9. REPORTE HTML DE COBERTURA

Al ejecutar `npm run test:coverage`, Jest genera automáticamente un reporte
visual en:
  coverage/lcov-report/index.html

Cómo abrirlo:
  Opción A: Doble clic en el archivo index.html desde el explorador.
  Opción B: En la terminal:
    start coverage/lcov-report/index.html     (Windows)
    open coverage/lcov-report/index.html      (macOS)
    xdg-open coverage/lcov-report/index.html  (Linux)

Interpretación de colores en el reporte HTML:

  VERDE  ── Código cubierto
    La línea o rama fue ejecutada durante al menos un test.
    Indica que ese comportamiento está siendo validado.

  ROJO   ── Código NO cubierto
    La línea nunca fue ejecutada durante los tests.
    Indica un camino que podría tener errores no detectados.

  AMARILLO ── Cobertura parcial de ramas
    La línea fue ejecutada, pero no todos los caminos del if/else fueron
    tomados. Por ejemplo: se probó el caso `true` de un if, pero no el `false`.

El reporte muestra cada archivo de forma individual, con las líneas
coloreadas y un contador de cuántas veces fue ejecutada cada línea.
